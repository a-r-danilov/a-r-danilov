# Проект: доработка пайплайна для аналитики продаж и клиентской возвращаемости


## Общая цель

Адаптировать существующий пайплайн для:
1. Учёта новых статусов заказов (`shipped` / `refunded`) в витрине `mart.f_sales`.
2. Расчёта метрик клиентской возвращаемости (`customer retention`) и загрузки в витрину `mart.f_customer_retention`.


---

## Этап 1. Учёт статусов заказов в `mart.f_sales`


**Задача:** обновить пайплайн, чтобы витрина корректно обрабатывала:
- статус `shipped` (доставка);
- статус `refunded` (возврат средств).

**Исходные данные:**
- Новые инкременты поступают через API.
- Каждый заказ имеет статус: `shipped` или `refunded`.


**Действия пайплайна:**
1. **Инициализация отчёта** через `POST /generate_report`:
   - Передача заголовков: `X-Nickname`, `X-Cohort`, `X-Project`, `X-API-KEY`.
   - Получение `task_id` для отслеживания статуса.
2. **Получение отчёта** через `GET /get_report`:
   - Ожидаем статус `SUCCESS` и получаем `report_id`.
   - Загружаем 4 файла из S3:  
     - `customer_research.csv`  
     - `user_order_log.csv`  
     - `user_activity_log.csv`  
     - `price_log.csv`.
3. **Загрузка инкрементов** через `GET /get_increment`:
   - Указываем `report_id` и дату в формате `YYYY-MM-DDTHH:MM:SS`.
   - Получаем `increment_id` и загружаем 3 файла:  
     - `customer_research_inc.csv`  
     - `user_order_log_inc.csv`  
     - `user_activity_log_inc.csv`.
4. **Обработка статусов**:
   - Все существующие данные в `mart.f_sales` считаются `shipped`.
   - Новые записи с `refunded` учитываются отдельно (возврат).
5. **Обеспечение backward compatibility**:
   - Сохранение совместимости с предыдущими версиями витрины.

---

## Этап 2. Расчёт клиентской возвращаемости (`customer retention`)

**Задача:** создать витрину `mart.f_customer_retention` для анализа:
- возвращаемости клиентов по неделям;
- дохода и возвратов в разрезе категорий товаров.


**Метрики витрины:**
1. `new_customers_count` — новые клиенты (1 заказ за неделю).  
2. `returning_customers_count` — вернувшиеся клиенты (>1 заказа за неделю).  
3. `refunded_customer_count` — клиенты с возвратами за неделю.  
4. `period_name` — период (`weekly`).  
5. `period_id` — идентификатор периода (номер недели).  
6. `item_id` — идентификатор категории товара.  
7. `new_customers_revenue` — доход от новых клиентов.  
8. `returning_customers_revenue` — доход от вернувшихся клиентов.  
9. `customers_refunded` — количество возвратов.


**Действия пайплайна:**
1. **Сбор данных** из:
   - Основного отчёта (`user_order_log.csv`);
   - Инкрементов (`user_order_log_inc.csv`).
2. **Агрегация по неделям**:
   - Группировка заказов по `customer_id` и `item_id`.
   - Расчёт количества заказов на клиента за неделю.
3. **Классификация клиентов**:
   - `new` — 1 заказ за неделю;
   - `returning` — >1 заказа за неделю;
   - `refunded` — клиенты с возвратами.
4. **Расчёт доходов и возвратов**:
   - Суммирование выручки по категориям клиентов.
   - Подсчёт количества возвратов.
5. **Загрузка в витрину** `mart.f_customer_retention`:
   - Вставка данных с указанием `period_id` (номер недели) и `item_id`.


---

## Ожидаемый результат


1. **Обновлённая витрина `mart.f_sales`**:
   - Корректный учёт статусов `shipped` и `refunded`.
   - Совместимость с предыдущими версиями данных.

2. **Новая витрина `mart.f_customer_retention`**:
   - Еженедельные метрики возвращаемости клиентов.
   - Разбивка по категориям товаров (`item_id`).
   - Данные для анализа эффективности категорий и программ лояльности.

3. **Доработанный пайплайн**:
   - Автоматизированная загрузка данных через API.
   - Обработка инкрементов.
   - Регулярное обновление витрин.


---


## Технические детали


- **API-эндпоинты**:  
  - `/generate_report` (POST)  
  - `/get_report` (GET)  
  - `/get_increment` (GET)
- **Хранение данных**: Yandex Cloud S3 (`storage.yandexcloud.net`).
- **Формат дат**: `YYYY-MM-DDTHH:MM:SS` (например, `2020-01-22T00:00:00`).
- **Заголовки API**:  
  - `X-Nickname`  
  - `X-Cohort`  
  - `X-Project`  
  - `X-API-KEY`
